# DECISIONS.md - Technical Decisions

## 2026-03-15

### DECISION-001: Workspace structure
- Decision: use a monorepo with pnpm workspace for frontend packages and Cargo workspace for Rust crates.
- Status: active

### DECISION-002: Current local data layer
- Decision: keep `rusqlite` as the temporary local data capability while DuckDB and Polars remain blocked by dependency conflicts.
- Status: active

### DECISION-003: CSV parsing strategy
- Decision: use the Rust `csv` crate for `DataSourceLoader` instead of the temporary std-only parser.
- Why: the environment can now download crates from `crates.io`, and the crate is safer and more maintainable than a handwritten parser.
- Status: active

### DECISION-004: Project export format
- Decision: implement the first export flow as a JSON project snapshot generated by a Tauri command.
- Why: the current product state does not yet have query results or dashboards to export, so a JSON snapshot provides a usable artifact without introducing a premature reporting/export architecture.
- Status: active

### DECISION-005: Initial query engine scope
- Decision: implement the first `QueryEngine` against CSV data sources inside `analytics-core` instead of waiting for DuckDB or Polars.
- Why: this satisfies the semantic-query execution milestone and unblocks command/UI integration while the intended analytics stack remains blocked by dependency conflicts.
- Status: active

### DECISION-006: Frontend query validation step
- Decision: expose query execution first through a sample-query validation flow in the existing screen before building the full query builder.
- Why: this proves the Tauri command contract end to end with minimal UI churn and keeps the next task focused on the actual query-builder interaction design.
- Status: active

### DECISION-007: Query builder scope
- Decision: implement the first query builder inside the current validation screen instead of introducing a new route or dashboard shell.
- Why: the app still lacks the base layout primitives targeted by `TASK-1.4.2`, so the lowest-risk way to satisfy the interaction milestone is to keep the builder local and defer broader page structure changes to the layout task.
- Status: active

### DECISION-008: Initial layout primitive scope
- Decision: introduce a minimal reusable layout layer in `apps/desktop/src/components` for app shell, panels, and section headers before moving to charts and dashboards.
- Why: this reduces structural duplication in the current validation screen and provides a stable frontend skeleton without prematurely forcing the long-term feature-folder architecture described in `AGENT.md`.
- Status: active

### DECISION-009: Initial chart library scope
- Decision: implement the first chart library layer as a reusable query-result chart component with bar and line variants, backed by `echarts-for-react`.
- Why: this satisfies the visualization milestone with the smallest reusable surface area and lets dashboard work build on a concrete chart abstraction instead of raw table results.
- Status: active

### DECISION-010: Dashboard delivery shape
- Decision: deliver the first dashboard as a dedicated app mode within the existing desktop screen instead of introducing route-level navigation.
- Why: this keeps the new dashboard aligned with the current single-window product shape, reuses the existing shell/layout primitives, and avoids premature routing structure while still satisfying the dashboard milestone.
- Status: active

### DECISION-011: AppError shape
- Decision: standardize desktop backend command failures on a structured `AppError` object with `code`, `message`, and optional `details`.
- Why: this gives the frontend a stable contract for readable user-facing errors and later telemetry/logging, while reducing repetitive string mapping in Tauri command handlers.
- Status: active

### DECISION-012: Backend logging format
- Decision: emit backend command logs as JSON-line entries through `env_logger` instead of plain text messages.
- Why: the hardening phase needs logs that are machine-readable and consistent across command start, success, and failure paths without introducing a larger observability dependency.
- Status: active

### DECISION-013: Frontend logging shape
- Decision: collect frontend interaction logs as JSON-line console entries backed by a small in-memory rolling buffer instead of introducing a dedicated telemetry service or persistence layer.
- Why: this satisfies the immediate hardening requirement to record key UI interactions while keeping the implementation dependency-free and aligned with the current offline desktop product scope.
- Status: active

### DECISION-014: Frontend E2E execution mode
- Decision: run the first frontend E2E suite in browser mode through Playwright with a mockable Tauri bridge instead of requiring native Tauri shell orchestration in tests.
- Why: this keeps the E2E path deterministic and CI-friendly while still exercising the real React UI and command wiring for the critical user flows.
- Status: active

### DECISION-015: Frontend state-management baseline
- Decision: use React Query for frontend command-backed server state and Zustand for lightweight UI state such as view mode, chart variant, and active selection.
- Why: this matches the target architecture in `AGENT.md` while avoiding a premature wholesale rewrite of the current screen into a larger state container.
- Status: active

### DECISION-016: Restore analytics dependencies before pipeline migration
- Decision: re-enable DuckDB and Polars in `analytics-core` with a dedicated stack-verification path before migrating ingestion and query execution onto the new data path.
- Why: this isolates dependency and platform-linking problems first, keeps current CSV-based behavior stable, and gives the next ingestion task a verified analytics foundation.
- Status: active

### DECISION-017: Preserve CSV query inputs during cache ingestion
- Decision: keep `DataSourceInfo.path` pointed at the original CSV source while adding an optional `cache_path` for the generated Parquet artifact.
- Why: this allows Parquet ingestion to ship without breaking the current CSV-backed query flow, and gives `TASK-4.3.3` an explicit handoff point for the later DuckDB query-path switch.
- Status: active

### DECISION-018: Controlled DuckDB query planning for Parquet execution
- Decision: execute Parquet-backed semantic queries through a constrained DuckDB SQL planner that only interpolates validated identifiers and binds runtime values as parameters.
- Why: the current DuckDB crate does not expose a higher-level relation API sufficient for this semantic-query surface, so a tightly controlled planner is the smallest practical way to switch execution onto Parquet while keeping value binding safe.
- Status: active

### DECISION-019: Move source loading boundary into connectors
- Decision: move CSV loading, preview/schema inference, and Parquet cache materialization into `crates/connectors`, while keeping `analytics-core` focused on semantic query execution.
- Why: this aligns the implemented runtime boundary with `AGENT.md`, removes the temporary loader/query coupling inside `analytics-core`, and creates a clean expansion point for later Excel, Parquet, and local database adapters.
- Status: active

### DECISION-020: Move snapshot export boundary into export-runtime
- Decision: move the JSON project snapshot payload and write logic into `crates/export-runtime`, while keeping the Tauri export command in the desktop crate as the frontend-facing boundary.
- Why: this aligns the export implementation with `AGENT.md` using the smallest safe migration, preserves the existing command contract, and creates a clean base for later CSV/XLSX/PDF export expansion.
- Status: active

### DECISION-021: Move backend logging and command timing into telemetry and job-runner
- Decision: move structured backend log formatting and emission into `crates/telemetry`, and move reusable synchronous command timing into `crates/job-runner`, while keeping Tauri commands in the desktop crate.
- Why: this activates the remaining scaffolded backend support crates with a concrete implementation path, removes duplicated timing/logging code from the desktop crate, and aligns the runtime boundaries with `AGENT.md` without introducing asynchronous job orchestration prematurely.
- Status: active

### DECISION-022: Migrate frontend structure incrementally by relocating live modules first
- Decision: move the currently active frontend runtime files into `app`, `pages`, `features`, `shared`, and `entities` with compatibility glue, instead of attempting a full behavioral rewrite during the structure migration task.
- Why: this satisfies the architectural direction in `AGENT.md` with lower risk, keeps the current desktop behavior stable, and creates a workable base for the next UI-layer migration and later shared package extraction.
- Status: active

### DECISION-023: Adopt Tailwind and shadcn-style primitives incrementally on the existing screen
- Decision: establish the UI-layer migration through a Tailwind global stylesheet, a `components.json` shadcn-style baseline, and a small shared primitive set in `src/shared/ui`, instead of attempting a full component-library swap in one pass.
- Why: this meets the `AGENT.md` UI-direction requirement with bounded risk, keeps current frontend behavior stable, and creates clean extraction targets for the later `ui-kit` and `chart-presets` package activation task.
- Status: active

### DECISION-024: Activate frontend packages through package entrypoints before standalone package builds
- Decision: activate `packages/ui-kit` and `packages/chart-presets` by adding real package manifests and entrypoints, and point those entrypoints at the current shared frontend runtime implementations before introducing independent package build tooling.
- Why: this satisfies the package-consumption requirement in `AGENT.md` with lower migration risk, lets the desktop app switch to package imports immediately, and preserves momentum toward the remaining backend-alignment work.
- Status: active

### DECISION-025: Prefer selective ECharts core imports before more chunk tuning
- Decision: address the current chart bundle-size warning first by switching the chart renderer from the default `echarts-for-react` entrypoint to `echarts-for-react/lib/core` with explicit `echarts/core` module registration for only the bar/line charts, grid, tooltip, and renderer primitives the product currently uses.
- Why: local dependency inspection shows the default `echarts-for-react` entrypoint initializes against the full `echarts` package, while the library's documented `core` path supports selective registration. That is a clearer size-reduction lever than additional `manualChunks` tuning alone, because the current warning is driven by too much chart code being included in the lazy `charts` chunk in the first place.
- Status: active

### DECISION-026: Accept the residual lazy chart chunk size with an explicit warning limit
- Decision: after reducing the async chart bundle from 1,060.77 kB to 521.44 kB, accept the remaining size as an intentional tradeoff for the current bar/line chart experience and raise Vite's `chunkSizeWarningLimit` to 550 kB.
- Why: the remaining overrun is small, the chart bundle is already lazy-loaded behind the dashboard view, and further splitting would add complexity for limited near-term value. Encoding the accepted threshold in Vite keeps build output focused on materially new regressions instead of repeating a known, reviewed warning.
- Status: active

### DECISION-027: Persist multi-view dashboards with legacy single-view compatibility
- Decision: extend project metadata with `dashboard_views` and `active_dashboard_view_id`, but continue reading the older single `dashboard_state` JSON shape as a compatibility fallback.
- Why: this unlocks multiple saved dashboard views without breaking previously saved projects or exported snapshots that only contain one dashboard payload.
- Status: active

### DECISION-028: Model dashboard layout editing as ordered sections with width state
- Decision: persist dashboard layout editing as an ordered `sections` array containing section ids and width choices, instead of introducing a heavier freeform grid model at this stage.
- Why: the current product only needs editable ordering and width for the fixed chart/query/table sections, so a lighter metadata model satisfies the acceptance criteria while keeping restore compatibility and UI complexity bounded.
- Status: active

### DECISION-029: Reuse one query-result table renderer across workspace and dashboard

### DECISION-030: Keep the job activity panel session-scoped and fed by polled job snapshots

### DECISION-031: Keep desktop E2E import/export coverage in browser mode with a stateful mocked Tauri bridge

### DECISION-032: Use ESLint plus Rust formatting checks as the first enforced lint baseline

### DECISION-033: Accept the current Tailwind-based ui-kit as a documented UI-stack exception

### DECISION-034: Run cached-query performance regression checks through an ignored release-path integration test
- Decision: implement the first cached analytics performance guard as an ignored integration test wrapped by `scripts/dev/run-cached-query-perf.ps1`, and run it with the `stable-x86_64-pc-windows-msvc` toolchain in release mode.

### DECISION-035: Use data-source loading as the first non-export tracked job path
- Decision: expand `job-runner` adoption by moving the desktop `load_data_source` workflow onto tracked start/poll/take commands, while keeping the older synchronous command available as a compatibility path.
- Why: data-source loading is the lowest-risk non-export workflow that already crosses the connector, cache materialization, and workbench hydration boundary, so it satisfies the AGENT.md job-lifecycle requirement without forcing a broader import/export redesign in one round.

### DECISION-036: Standardize Windows Rust validation through one MSVC wrapper and keep local workspace validation on `cargo check`
- Decision: route Windows repository validation through a shared `invoke-rust-command.ps1` wrapper that defaults to `stable-x86_64-pc-windows-msvc`, and keep `validate-workspace.ps1` on lighter `cargo check` commands for the heavy crates instead of forcing local link-heavy `cargo test --no-run` steps.
- Why: this removes the incorrect GNU default from local scripts and docs, while also keeping the repository-wide validation helper usable on constrained Windows machines where full local link steps can still fail for environmental reasons unrelated to the code changes under review.
- Status: active

### DECISION-037: Isolate Windows workspace validation artifacts from the default Cargo target tree
- Decision: run `validate-workspace.ps1` Rust checks through a dedicated `target/validation-msvc-check` directory with incremental compilation disabled and `-C debuginfo=0`, and automatically clear that cache when it grows past 6 GB.
- Why: the reviewed MSVC path still accumulated enough metadata, PDB, and intermediate artifacts in the shared `target` tree to trigger avoidable local `os error 112` failures. A validation-only target cache keeps repository checks from inheriting unrelated build output and provides one bounded cleanup point.
- Status: active

### DECISION-038: Prioritize AGENT.md direct-SQL closure before the remaining compliance tasks
- Decision: reopen the AGENT.md alignment backlog and make removal of direct SQL string construction in the Parquet query path the next single task before tackling UI-library and telemetry-depth follow-ups.
- Why: the strict audit found that the current planner and executor still use `format!` and `push_str` to assemble executable SQL in production code, which is a direct violation of an explicit AGENT.md prohibition and therefore a higher-priority compliance gap than the remaining architectural refinements.
- Status: active

### DECISION-039: Move the live desktop primitive layer into package-local shadcn-style implementations and use Radix for Select
- Decision: keep the existing workbench layout surfaces stable, but replace the primitive exports in `packages/ui-kit` with package-local shadcn-style implementations and migrate the active select inputs onto `@radix-ui/react-select`.
- Why: this satisfies the literal AGENT.md UI requirement with bounded migration risk. The workbench only needed a small set of primitives immediately, and Select was the control with the clearest accessibility and interaction gain from Radix-backed behavior.

### DECISION-040: Use cache warmup as the first non-import/export tracked maintenance workflow
- Decision: satisfy the broader AGENT.md job-runner requirement by introducing a first-class tracked cache-warmup workflow for active data sources, rather than waiting for a larger profiling subsystem.
- Why: the desktop already stores or directly uses Parquet artifacts, so cache warmup can reuse existing loader/runtime boundaries with lower churn than a new profiling pipeline while still proving tracked job lifecycle, cancellation, and UI surfacing outside import/export.

### DECISION-041: Implement telemetry depth through in-memory command metrics and a runtime debug toggle
- Decision: close the remaining telemetry gap with an in-memory command-metrics registry inside `crates/telemetry`, plus an explicit debug-logging control that can be set from environment variables and changed at runtime through the desktop backend.
- Why: this satisfies the AGENT.md expectation of local metrics sampling and debug controls without introducing a heavier persistence or UI surface. The existing command logging path already carries command names and durations, so a local registry and runtime toggle provide the smallest defensible telemetry expansion.
- Status: active

### DECISION-042: Treat `cargo test -p desktop --lib` as an optional deep validation step, not the default local smoke path
- Decision: add a dedicated `run-desktop-light-validation.ps1` wrapper for routine Windows/MSVC desktop backend validation, and keep the heavier `cargo test -p desktop --lib` path documented as an explicit escalation step.
- Why: the desktop crate's Tauri test target is materially more expensive to link than the reusable backend crates, so defaulting everyone onto `desktop --lib` tests slows routine local feedback without adding proportional signal for every backend change.
- Status: active

### DECISION-043: Reopen AGENT parity work around the documented DuckDB cache path before starting unrelated feature work
- Decision: treat restoration of a DuckDB-backed cached-Parquet aggregation path as the current single task, and queue tracked profiling-job coverage immediately behind it.
- Why: the 2026-03-17 strict re-audit found that the live executor in `crates/analytics-core/src/executor/mod.rs` no longer matches the `AGENT.md` cache-query architecture, which is a more direct product-spec drift than starting a new unrelated feature.
- Status: active

### DECISION-044: Choose DuckDB as the target cached-Parquet production executor after benchmark review
- Decision: use the new benchmark harness to compare DuckDB and the current Polars executor on the same cached Parquet aggregate query, and treat DuckDB as the target production path for larger offline workloads because it scales better in the local evidence collected so far.
- Why: the first successful local MSVC debug-profile results showed Polars only slightly ahead at 100k rows, while DuckDB was materially faster at 250k and 500k rows. Given the product goal of million-row offline analysis, the scaling trend matters more than the smallest-sample lead.
- Status: active

### DECISION-045: Restore the production cached-Parquet path with a constrained DuckDB SQL renderer
- Decision: switch the live cached-Parquet executor back to DuckDB, but keep the query surface constrained to SQL generated from the validated query plan and bind runtime filter plus pagination values as statement parameters.
- Why: the benchmark evidence favored DuckDB for larger offline workloads, but the repository still needed a bounded implementation that did not reopen the earlier free-form SQL-concatenation problem. A constrained renderer over validated identifiers and bound runtime values is the smallest practical production path supported by the current `duckdb` crate.
- Status: active

### DECISION-046: Finish AGENT parity on a DuckDB-only analytics-core path
- Decision: complete the remaining profiling-job parity work by implementing profiling on DuckDB-backed cached Parquet sources and remove the last Polars execution fallback from `analytics-core`.
- Why: once the benchmark-backed production choice was made, leaving a second dormant execution engine in the same crate only preserved drift risk and maintenance cost. The profiling workflow also naturally fits the same cached-Parquet boundary, so one DuckDB-only implementation closes both the runtime and architecture gaps cleanly.
- Status: active

### DECISION-047: Treat the remaining direct DuckDB SQL renderer as an explicit literal-compliance gap
- Decision: reopen the backlog after `TASK-10.2.1` and make removal of the remaining DuckDB SQL string renderer the next single task, ahead of additional feature work.
- Why: the 2026-03-17 follow-up audit confirmed that runtime values are parameterized and the execution surface is constrained, but `analytics-core` still assembles executable SQL strings in production code. Under a strict reading of `AGENT.md`, that is still a higher-priority compliance issue than broader validation polish.
- Status: active

### DECISION-048: Centralize DuckDB SQL generation behind a constrained AST boundary
- Decision: keep DuckDB as the execution engine, but move all remaining query construction for cached-Parquet execution and profiling into one internal `duckdb_sql` AST-plus-renderer module instead of continuing to assemble SQL fragments inline inside production workflows.
- Why: the current `duckdb` crate still does not expose a practical relation/expression builder for this semantic-query surface, so a centralized constrained renderer is the smallest workable boundary that removes ad hoc SQL construction from the execution and profiling paths while preserving parameter binding and regression coverage.
- Status: active

### DECISION-049: Normalize empty shared Select values through an internal sentinel
- Decision: keep the shared Radix-backed `packages/ui-kit/src/Select.tsx` API accepting `""` as the public "no selection" state, but translate that value to a non-empty internal sentinel before rendering `Select.Item` entries.
- Why: the new profiling Playwright coverage exposed a live frontend crash after data-source load because Radix Select rejects empty-string item values. Mapping the placeholder state internally preserves the existing query-builder/filter-bar API while matching Radix's runtime constraint.
- Status: active

### DECISION-050: Treat root tests/e2e parity as the next literal AGENT.md closure item
- Decision: after the latest strict re-audit, defer generic performance/stability cleanup until the repository either moves or explicitly wires the active desktop Playwright suite through the root `tests/e2e` scaffold described in `AGENT.md`.
- Why: the shipped feature set is now aligned, but the root `tests/e2e` directory remains a placeholder while the real suite lives under `apps/desktop/e2e` and CI points there directly. Under a strict reading of the documented repository layout, that is still a smaller but concrete structure mismatch that should be closed before switching to non-spec-driven polish work.
- Status: active

### DECISION-051: Standardize desktop browser regression checks behind one cleanup-aware wrapper
- Decision: after closing the root `tests/e2e` parity gap, codify desktop frontend browser validation as `scripts/dev/run-desktop-browser-validation.ps1` plus the root `pnpm validate:desktop-browser` entrypoint, instead of relying on contributors to remember the exact typecheck and Playwright flags manually.
- Why: the current desktop browser path is now important enough to deserve one stable validation command. The wrapper preserves the reviewed timeout/worker settings, reuses the migrated root `tests/e2e` suite through the existing desktop Playwright config, and cleans transient Playwright output directories by default so repeated local runs stay predictable.
- Status: active
